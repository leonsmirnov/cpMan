## Security — High

### S-H1. Filter concealed / transient pasteboard types
- **Where:** `Sources/cpMan/Services/ClipboardMonitor.swift` →
  `checkPasteboard()`.
- **Problem:** Every plain-text payload is captured regardless of the hint
  types the source app declares. Password managers (1Password, Bitwarden,
  iCloud Keychain) and many browsers mark sensitive copies with the
  conventional NSPasteboard hint types
  `org.nspasteboard.ConcealedType` / `org.nspasteboard.TransientType` /
  `org.nspasteboard.AutoGeneratedType`. cpMan does not honour those hints,
  so passwords can land in `history.json` in plaintext.
- **Fix:**
  1. Inspect `pasteboard.types` and skip recording when any of the three
     hint UTIs are present.
  2. Honour Universal Clipboard's `com.apple.is-sensitive` the same way.
  3. Extend `ClipboardMonitorPredicateTests` with cases that exercise each
     hint type and assert the predicate returns `false`.
- **References:**
  - <https://nspasteboard.org/>
  - `NSPasteboard.PasteboardType.transient` / `.concealed` (the
    well-documented private constants).

## Security — Medium

### S-M1. Preview directory permission race
- **Where:** `Sources/cpMan/UI/Picker/PickerView.swift` →
  `previewDirectory()`.
- **Problem:** `createDirectory(...)` runs first, then a separate
  `setAttributes([.posixPermissions: 0o700], ...)`. There is a small
  window during which the directory is world-readable (whatever the
  process umask gives us) before the chmod lands.
- **Fix:** Pass the POSIX mode through the create call directly:
  `createDirectory(at:withIntermediateDirectories:attributes:)` with
  `attributes: [.posixPermissions: 0o700]`. Only fall back to a chmod for
  already-existing directories.

### S-M2. Preview files written non-atomically
- **Where:** `PickerView.previewRow(_:)` — writes the temp file then chmods
  to `0o600`.
- **Problem:** Same race as S-M1 applies to the file itself.
  Additionally, a partial-write failure leaves a malformed file that the
  cleanup pruner will retain for up to 5 minutes.
- **Fix:** Write to `<dir>/.tmp.<uuid>`, chmod to `0o600`, then
  atomic-rename to the final name. Wrap in `try`/`catch` and `removeItem`
  on failure so we never leak a partial file.

## Security — Low

### S-L1. Symlink following in the preview-directory cleanup
- **Where:** Cleanup loop in `PickerView.previewDirectory()`.
- **Problem:** We call `removeItem(at:)` on every entry older than the
  cutoff. If an attacker (or careless user) plants a symlink in our
  scratch directory pointing at a file they don't own, we currently only
  remove the link itself — but we should be defensive against future
  refactors that might enumerate followed symlinks.
- **Fix:** Use `URLResourceValues` to read `isSymbolicLink` and
  `isRegularFile` and only remove regular files we own. Consider passing
  `[.skipsSubdirectoryDescendants, .skipsPackageDescendants]` to the
  directory enumerator.

## App Store submission — Accessibility justification (master branch only)

> The items below apply to the **master branch** build, which will reintroduce
> Accessibility (auto-paste, synthetic ⌘V via `CGEvent.postToPid`). The
> current branch has no AX surface and these tasks are not needed here.

### AS-1. "Why does your app need control of other applications?"
- **Deliverable:** 1–2 paragraph written answer, pasted into **App Store
  Connect → App Review Information → Notes** and reused in any individual
  reviewer reply.
- **Key points to cover:**
  1. The app does NOT read content from other apps, send Apple Events, or
     automate UI. It synthesises a **single ⌘V keystroke** into the
     previously focused window after the user has **explicitly chosen** an
     item in the picker.
  2. The feature is **opt-in** (see acceptance below — recommend defaulting
     it OFF for the first master submission so reviewers can confirm
     correctness without granting AX).
  3. The app **fully works without the entitlement**: selection just copies
     to the clipboard and the user presses ⌘V themselves.
  4. Without AX, macOS does not permit synthetic keystrokes from a
     sandboxed process, so there is no alternative API.

### AS-2. Demonstration video
- **Deliverable:** Screen recording (≤ 60 s, 1080p, no narration required)
  saved to `Documentation/AppStoreReviewAssets/auto-paste-demo.mp4` on
  master (or hosted privately and linked from the review notes).
- **Storyboard the reviewer will follow:**
  1. Launch the app → menu-bar clipboard icon visible (no Dock entry).
  2. Open Settings → **General** → show the **Behaviour → "Auto-paste on
     selection"** toggle (ON for this segment of the demo).
  3. Open the **Accessibility Permission** section, click **Open Settings…**,
     toggle the app ON in *System Settings → Privacy & Security →
     Accessibility*, return to the app.
  4. Copy a paragraph in Safari/Notes.
  5. Switch to TextEdit, press **⌃⌥V** → picker appears → press Return on
     the highlighted item → app focus returns to TextEdit and the paragraph
     is pasted.
  6. Re-open Settings → flip auto-paste OFF → repeat the picker sequence →
     show that the panel now only copies (user presses ⌘V manually). This
     proves the entitlement is the **only** reason the app needs to control
     another application.

### AS-4. Keep the four AX surfaces working (master maintenance burden)
- **Background:** When master ships with Accessibility, four interlocking
  surfaces all have to behave correctly. None of them is unit-testable
  end-to-end because TCC is system-level state — regressions silently make
  the app look broken to reviewers and end-users.
- **The four surfaces:**
  1. **AX prompt flow** — `AccessibilityService.openAccessibilitySettings()`
     deep-link + `AXIsProcessTrustedWithOptions` registration in
     `AppDelegate.applicationDidFinishLaunching`.
  2. **Settings toggle + status row** — Auto-paste toggle, green/red AX
     state badge, "Open Settings…" button. Depends on
     `AccessibilityService` polling every second.
  3. **Picker "AX denied" banner** — orange strip with the in-context
     "Open Settings" affordance.
  4. **Copy-only fallback** — `PasteService.paste(item:into:)` must still
     write the item to the pasteboard when AX is not granted so the
     picker is useful for users who decline.
- **Regression checklist (after any change touching the surfaces above
  AND before every master-branch App Store submission):**
  - [ ] Clean Mac (or fresh user account / VM): no prior TCC grant.
  - [ ] Install master build → menu-bar icon visible.
  - [ ] Open picker (⌃⌥V) **without** granting AX → orange banner appears.
  - [ ] Click "Open Settings" in the banner → System Settings opens at
        Privacy & Security → Accessibility.
  - [ ] Toggle cpMan ON → return to the app → banner disappears within
        ~2 s.
  - [ ] Settings → General → Accessibility Permission row turns green ✓.
  - [ ] Copy text in Safari → select it in the picker → focus returns to
        Safari and the text pastes (auto-paste worked).
  - [ ] Toggle "Auto-paste on selection" OFF → repeat → text only copies,
        no synthetic ⌘V (copy-only fallback works).
  - [ ] Re-toggle AX OFF in System Settings → re-open picker → banner
        re-appears without an app restart.
- **When to re-run:**
  - After every macOS minor update (TCC behaviour shifts between point
    releases — this is the #1 source of silent regressions).
  - After every refactor touching any of the four surfaces.
  - After every signing-identity change (ad-hoc → Developer ID →
    Apple Distribution): TCC keys grants by code signature, so the grant
    is lost and the prompt flow must work again.

### AS-5. Per-macOS-version compatibility log for the AX flow
- **What:** Maintain a small table at the top of
  `Documentation/AppStoreReviewAssets/ax-compat-log.md` (master branch)
  recording, for each macOS version we have tested:
  - macOS version + build number (e.g. `15.3 (24D60)`).
  - Whether `AXIsProcessTrustedWithOptions(prompt: true)` actually shows a
    dialog or is silent.
  - Whether the
    `x-apple.systempreferences:com.apple.preference.security?Privacy_Accessibility`
    deep-link still opens the correct pane.
  - Any extra steps the user has to take in that version (e.g. macOS 26
    moved the toggle under a new sub-section).
- **Why:** Apple silently changes TCC and System Settings layout between
  versions. Without a log we re-discover the same bugs every release.
